# ACCEPTED FINDINGS

## 1. Batch Management is the canonical UI benchmark

Accepted.

batch-management.html is now the official design reference.

Reasons:

* strongest UX consistency
* best dark mode implementation
* best responsive behavior
* shared token direction
* skeleton loading
* toast system
* polished cards/layout
* retry-safe patterns
* strongest operational feel

Future pages should align visually and architecturally with this page.

---

## 2. Multiple design systems exist

Accepted.

Current systems:

1. legacy custom CSS
2. Tailwind CDN pages
3. premium Manrope-based batch-management system

This causes:

* inconsistent UX
* inconsistent spacing
* inconsistent navs
* inconsistent typography
* duplicated components

Decision:
Standardize around the Batch Management design system.

---

## 3. Large inline JS/CSS is a maintenance risk

Accepted.

Large files:

* admin-portal.html
* admin-review.html
* registration-form.html
* applicant-directory.html

Issues:

* difficult testing
* duplicated utilities
* hard debugging
* fragile refactors

Decision:
Incrementally extract:

* API helpers
* UI helpers
* modals
* toasts
* layouts
* shell/navigation
* tables/cards

into shared modules.

---

## 4. Shared shell architecture is required

Accepted.

Current issue:
Each page independently recreates:

* nav
* topbar
* auth handling
* theme toggle
* layout
* spacing

Decision:
Use:

* teacher-shell.js
* future admin-shell.js
* shared layout.css/components.css

---

## 5. Dual backend mode is dangerous long-term

Accepted.

Current state:
Legacy archive + Supabase notes coexist.

Risk:

* invisible backend switching
* inconsistent auth
* inconsistent data flow
* hidden bugs
* migration confusion

Current temporary decision:
Maintain compatibility layer during migration only.

Long-term goal:
Supabase becomes sole source of truth.

---

## 6. Security hardening is now priority-level work

Accepted.

Current concerns:

* hardcoded anon keys
* client-side role checks
* unclear RLS coverage
* placeholder auth fallbacks

Decision:
Move toward:

* env injection
* strict RLS
* edge-function enforcement
* server-side authorization
* removal of committed credentials

# CANONICAL ENGINEERING PRINCIPLES

* Incremental migration only
* Preserve working flows
* Avoid giant rewrites
* Stability over speed
* Shared infrastructure preferred
* Retry-safe writes preferred
* Soft-delete/archive preferred
* Modular utilities preferred
